# Infra Dashboard 操作マニュアル

## このマニュアルについて

Infra Dashboardは、AWSアカウント内のEC2/RDSについて、現在の状態、日ごとの稼働時間、CPU使用率、過去の稼働履歴を1画面で確認するためのツールです。

このマニュアルは、デプロイ済みの画面を利用する人と、画面表示を運用に合わせて調整する管理者を対象にしています。AWSへのデプロイ方法は[デプロイ手順](instructions.md)を参照してください。

この画面はPCでの利用を前提としています。スマートフォン向けのレイアウト調整は行っていません。

---

## 1. 最初に行う操作

1. CloudFront URLをPCのブラウザで開きます。
2. 表示したい「インスタンス」「日付」「環境」「リージョン」を選びます。
3. 必要に応じて、Nameタグフィルター、アカウント、列ON/OFFを変更します。
4. 「表示・更新」を押します。
5. 各インスタンスの現在状態、起動時間、最大CPU、24時間バーを確認します。

URLに `loadnow` を付けると、ページを開いた直後に「表示・更新」が自動実行されます。

```text
https://xxxxxxxxxx.cloudfront.net/?date=today&loadnow
```

設定を変更しただけでは表が更新されない場合があります。「表示・更新」が表示されたら、そのボタンを押してください。

---

## 2. 一覧画面の操作

### インスタンス

表示するサービスを選択します。

- `EC2＆RDS`：EC2とRDSを同時表示
- `EC2`：EC2のみ表示
- `RDS`：RDSのみ表示

RDSを使用しない構成では、EC2だけが選択肢になります。

### 日付

次の方法で表示日を変更できます。

- 「<-1日」「+1日>」：前日または翌日へ移動
- 「今日」：本日に戻る
- 「N日前 mm/dd(曜)」：過去日を直接選択

選択できる範囲は最大730日前です。ただし、実際に表示できるのはデータ収集を開始した日以降です。

日付を変更すると自動的に再読込されます。

### 環境などの分類

初期設定では `Env` タグを「環境」として扱います。

例：

- 本番のみ
- 開発・検証
- 開発・検証・テスト
- 全て（タグなしを含む）

選択肢と対象タグは `config.js` で変更できます。詳細は「7. config.jsで画面表示を変更する」を参照してください。

### Nameタグフィルター

画面右上の「Nameタグフィルター」には正規表現を入力できます。入力後に「表示・更新」を押すと、Nameタグが正規表現に一致するインスタンスだけを表示します。

例：

| 入力 | 表示対象 |
|---|---|
| `.*` | すべて |
| `^prod-` | `prod-` で始まるNameタグ |
| `batch$` | `batch` で終わるNameタグ |
| `^(web|api)-` | `web-` または `api-` で始まるNameタグ |

大文字と小文字は区別されます。正規表現として不正な内容を入力すると、入力欄の背景が赤くなり、直前の有効な条件が引き続き使われます。

Nameタグフィルターの入力値は、現在のURLには保存されません。条件をURLで共有したい場合は、後述の `search` またはタグフィルターを使う方法も検討してください。

### リージョン

画面右上のリージョン名と「▼」を押すと、`config.js` のアカウント設定に登録されたリージョンを選択できます。

リージョンを変更したら「✓」で閉じ、「表示・更新」を押します。

### アカウント

複数アカウントが設定されている場合は、「アカウント：[▼]」から表示対象を選べます。複数のアカウントを同時に選ぶと、同じ表へまとめて表示します。

1つも選択しない状態では確定できません。

### 列ON/OFF

チェックボックスで次の列を表示・非表示にできます。

- アカウント
- 環境などの分類列
- Applicationタグなどのグループ列
- インスタンスタイプ
- VPC
- AZ
- 起動時間

選択状態はブラウザに保存され、次回アクセス時にも引き継がれます。

### CPU使用率

「CPU使用率(0-100%)」をONにすると、24時間バーの内側へCPU使用率を色で重ねます。

- 低いCPU使用率：緑系
- 高いCPU使用率：黄、橙、赤、桃系
- 「最大CPU%」：表示日の最大値

CPUデータを読み込むため、ONにすると表示完了まで少し時間がかかる場合があります。

24時間バー左側の小さなグラフアイコンを押すと、その日のCloudWatch CPUグラフを行の下へ表示します。63日以上前のグラフは、CloudWatchのデータ保持仕様に合わせて1時間単位になります。

### 検索、コピー、Excel出力

表の「検索」は、すでに読み込まれた表全体を絞り込みます。Nameタグだけでなく、アカウント名、タグ、インスタンスタイプ、状態なども検索対象になります。

Nameタグフィルターとの違いは次のとおりです。

| 機能 | 対象 | 再読込 |
|---|---|---|
| Nameタグフィルター | Nameタグのみ。正規表現 | 「表示・更新」が必要 |
| 表の検索 | 読み込み済みの表全体 | 即時反映 |

出力ボタンでは次の操作ができます。

- コピー：表をクリップボードへコピー
- Excel出力（表示列のみ）：現在表示している列だけを出力
- Excel出力（全ての列）：非表示列も含めて出力

Excelファイル名は `status_YYYYMMDD` 形式です。

## 3. 表示内容の見方

### 一覧の各列

| 列 | 意味 |
|---|---|
| Nameタグ | EC2のNameタグ。RDSはNameタグがない場合、DB識別子を表示 |
| アカウント | データを取得したAWSアカウント |
| 環境 | 初期設定ではEnvタグ。名称と対象タグは変更可能 |
| Applicationタグ | 初期設定ではApplicationタグ。名称と対象タグは変更可能 |
| InstanceType | EC2インスタンスタイプまたはRDSインスタンスクラス |
| VPC | VPC名。列ON/OFFで表示 |
| AZ | アベイラビリティゾーン。列ON/OFFで表示 |
| 稼働状況 | 現在のAWSステータス。取得日時が見出しに表示される |
| 起動～停止 | 表示日の起動・停止時刻 |
| 起動時間(h) | 表示日に稼働していた時間の合計 |
| 最大CPU% | 表示日の最大CPU使用率 |
| 0:00～24:00 | 稼働状態とCPU使用率を重ねた24時間バー |

### 24時間バー

24時間バーは、表示日の0:00から24:00までを横方向に表します。

| 表示 | 意味 |
|---|---|
| 薄緑 | 稼働中 |
| 灰色 | 停止中、またはRDSが通常稼働状態ではない時間 |
| `FU` / Storage-Full | RDSの空きストレージが不足した状態 |
| 「システム異常」 | EC2の基盤側ステータスチェックで異常が検出された状態 |
| `Te` / Terminated | EC2が削除された状態 |
| バー前方の黒系部分 | その時刻にはまだインスタンスが存在していない |
| バー後方の黒系部分 | その時刻以降、インスタンスが削除済み |
| バー内側の色 | CPU使用率 |

今日のバーは現在時刻まで描画され、未来の時間帯は描画されません。

状態は定期収集時点の結果です。短時間で発生し、次回収集前に解消した変化は記録されない場合があります。

### 起動～停止

例：

- `08:00～18:00`：8:00に起動し、18:00に停止
- `00:00～`：前日から継続して稼働中
- 複数行：1日の中で複数回起動・停止
- `＿:＿`：対象データがない、または状態を特定できない

### 現在の稼働状況

表の「取得」または「状況取得」を押すと、現在のAWSステータスを取得します。一覧の初回表示後にも自動取得されます。

取得に成功すると、列見出しに `月/日 時:分` が表示されます。これは現在状態を取得した時刻です。

#### EC2

`running`、`stopped` などAWSの現在状態を表示します。

`running` の下に `(2/3 status)` のような点滅表示が出た場合、AWSのステータスチェックがすべて正常ではありません。

- 分母2：インスタンスとシステムの2項目を確認
- 分母3：インスタンス、システム、接続されたEBSの3項目を確認
- 分子：正常だった項目数

例：

- `(2/3 status)`：3項目のうち2項目が正常
- `(0/2 status)`：2項目とも正常ではない

起動後10分以内は、初期化中の状態を異常扱いしないため、この補足表示を出しません。

点滅表示が続く場合は、AWSマネジメントコンソールのEC2ステータスチェック、システムログ、EBSの状態を確認してください。

#### RDS

`available`、`backing-up`、`modifying`、`storage-full`、`failed` など、AWSが返したDBインスタンスの状態をそのまま表示します。

`storage-full`、`failed`、`incompatible-*` などが点滅している場合は、AWSマネジメントコンソールでDBイベント、空きストレージ、設定状態を確認してください。

過去の24時間バーでは `storage-full` は「Storage-Full」として明示されます。それ以外の通常稼働ではないRDS状態は灰色で表示されます。

### 「システム異常」の意味

24時間バーの「システム異常」は、EC2の基盤側ステータスが `impaired` になった時間です。

OSやアプリケーション内部の異常を直接検出したものではありません。表示された場合は、AWSマネジメントコンソールの「システムステータスのチェック」とAWS Health Dashboardを確認してください。

一度記録された異常は、解消後も過去の履歴として残ります。

## 4. 警告とエラーの見方

### 「一部の稼働履歴を取得できなかったため、表示が正しくない可能性があります。」

EC2/RDS自体は見つかりましたが、その日の稼働履歴が空だった場合に表示されます。

この警告が出ている間は、24時間バー、起動～停止、起動時間を正しいものとして扱わないでください。

まず「表示・更新」を再実行します。解消しない場合は、管理者が次を確認してください。

- EventBridge Schedulerの `record-time` が有効か
- 稼働履歴収集Lambdaでエラーが出ていないか
- S3へ当日の履歴が保存されているか
- 対象リージョンが設定に含まれているか

### 「同一のNameタグで複数インスタンスが存在します。」

履歴画面で、同じNameタグを持つ複数のインスタンスが見つかった場合に表示されます。

赤い点線が付いたバーは、複数候補のうち代表となる1台を表示しています。リストア、再作成、Auto Scalingなどで同じNameタグが再利用された可能性があります。

この場合、Nameタグだけでは1台を一意に識別できません。上部のサーバー情報と対象日の前後を合わせて確認してください。

### 「サーバー情報取得不可」

履歴自体は残っていますが、現在情報や過去の保存情報から、インスタンスタイプ、Private IPなどを取得できなかった場合に表示されます。

削除から時間が経過している、保存開始前のデータである、対象リージョンやアカウントがURLと一致していない、といった可能性があります。

### 「日付が変わりました。ページを再読み込みしてください。」

画面を開いたまま日付が変わった場合に表示されます。ブラウザを再読み込みしてから操作を続けてください。

### 稼働状況が `-` になる

現在状態の取得に失敗したか、一覧のインスタンスとAWSの現在情報を対応付けられなかった可能性があります。

- 「状況取得」を再実行
- リージョンとアカウントを確認
- CloudFront/APIへのアクセスを確認
- 管理者がLambdaログを確認

## 5. 履歴画面

一覧画面のNameタグ横にある小さな履歴アイコンを押すと、そのNameタグの履歴画面が別タブで開きます。

### 上部のサーバー情報

Nameタグ部分を開くと、取得できた範囲で次の情報を表示します。

- Applicationタグなどのグループタグ
- EC2インスタンスID
- 環境などの分類タグ
- Private IP
- EC2インスタンスタイプ
- RDSインスタンスクラス

値にマウスを合わせると、コピー用のアイコンが表示されます。

### 表示期間

最初は今日を含む14日分を表示します。

- 「30日前まで表示」：31日分まで追加
- 「2ヶ月分を表示」：61日分まで追加
- 「↓ -2ヶ月」「↓↓ -6ヶ月」：表示基準を過去へ移動
- 「↑ +2ヶ月」「↑↑ +6ヶ月」：表示基準を現在へ戻す
- 「再読込」：現在の表示基準で再取得

表示基準は最大24ヶ月前まで移動できます。ただし、実際に確認できるのはデータが保存されている期間だけです。

### CPU使用率

履歴画面でも「CPU使用率」をON/OFFできます。ONにすると現在表示中の日付へCPU色を追加し、OFFにすると稼働状態だけのバーへ戻します。

各行のグラフアイコンから、その日のCloudWatch CPUグラフを表示できます。

## 6. URLのクエリ文字列

URLへクエリ文字列を付けると、表示条件を指定した状態で画面を開けます。

最初のパラメータは `?`、2つ目以降は `&` でつなぎます。

```text
index.html?date=today&region=ap-northeast-1&loadnow
```

空白、日本語、`%` などを含む値はURLエンコードしてください。

例としてNameタグ `%Batch server` は次のようになります。

```text
%25Batch%20server
```

### 一覧画面

| パラメータ | 値 | 画面への効果 |
|---|---|---|
| `date` | `today` / `yesterday` / `YYYYMMDD` | 表示日を指定 |
| `region` | `ap-northeast-1` など | 表示リージョンを指定 |
| `svcIdx` | `0`、`1`、`2` | サービス選択の何番目を使うか指定 |
| `categoryIdx` | `0`～`4` | 分類選択の何番目を使うか指定 |
| `account` | `0`、`1`、… | `config.js` の `accounts` 配列を0始まりで指定 |
| `accounts` | アカウント表示名のカンマ区切り | 一覧へ合成表示するアカウントを指定 |
| `application` | タグ値、カンマ区切り、`ALL` | Applicationタグで絞り込み。パラメータ名は変更可能 |
| `search` | 検索文字 | 表の検索欄へ初期値を設定 |
| `cpuutil` | `1` / `0` | CPU使用率を表示 / 非表示 |
| `pagelength` | `10`、`20`、`50`、`100` など | 1ページの表示件数 |
| `loadnow` | 値なし | ページ表示直後に「表示・更新」を自動実行 |

`svcIdx` は画面上の選択肢を0から数えます。RDSを使用する構成では通常、`0=EC2＆RDS`、`1=EC2`、`2=RDS` です。EC2だけの構成では `0=EC2` です。

`categoryIdx` も `config.js` の `categoryTag.options` を上から0、1、2…と数えます。

`pagelength` は `tablePageLengthOptions` に登録された値だけが有効です。

`date` の `YYYYMMDD` 形式は2039年まで対応しています。

URLで指定した `svcIdx`、`categoryIdx`、`cpuutil` は、ブラウザに保存された前回値より優先されます。

#### 一覧画面の例

今日の東京リージョンを開き、自動で表示します。

```text
index.html?date=today&region=ap-northeast-1&svcIdx=0&categoryIdx=0&cpuutil=1&loadnow
```

昨日のApplicationタグ `Batch` だけを表示し、表を `failed` で検索します。

```text
index.html?date=yesterday&application=Batch&search=failed&pagelength=50&loadnow
```

`application` は初期設定でのパラメータ名です。`groupTagFilter.keyURL` を変更した場合は、その名前を使用してください。

現行版では、`application` などのタグ値フィルターは英数字、`_`、`-` を使った値での利用を推奨します。日本語や空白を含むタグ値は、URLエンコードしても一致しない場合があります。

また、タグ値フィルターを指定すると、そのタグ自体を持たないリソースは除外されます。

### 履歴画面

| パラメータ | 値 | 画面への効果 |
|---|---|---|
| `account` | `0`、`1`、… | データ取得元アカウントを0始まりで指定 |
| `svc` | `ec2` / `rds` | 対象サービスを指定 |
| `region` | リージョンID | 対象リージョンを指定 |
| `name` | NameタグまたはRDS名 | 表示対象を指定 |
| `cpuutil` | `1` / `0` | CPU使用率を表示 / 非表示 |

通常は一覧画面の履歴アイコンから開くため、手動でURLを組み立てる必要はありません。

Nameタグ `%Batch server` のEC2履歴を開く例：

```text
history.html?account=0&svc=ec2&region=ap-northeast-1&name=%25Batch%20server&cpuutil=1
```

`name` がない場合、対象を特定できないため履歴を表示できません。

### URL共有時の注意

- 共有前に、URLを新しいタブで開いて期待どおり復元されるか確認してください。
- Nameタグフィルター入力欄の正規表現はURLへ保存されません。
- 表の検索文字を変更した直後は、アドレスバーの `search` がまだ更新されていない場合があります。確実に共有する場合はURLの `search` を確認してください。
- 複数アカウント構成では、`accounts` は `accountName` のカンマ区切りです。

## 7. config.jsで画面表示を変更する

設定ファイルは次の場所にあります。

```text
src/web/script/config.js
```

デプロイ済み環境ではS3の次の場所にあります。

```text
/web/script/config.js
```

ソース側を変更した場合は再デプロイが必要です。S3上のファイルを直接変更した場合は再デプロイ不要ですが、次回デプロイで上書きされるため、恒久変更はソースまたはデプロイオプションへ反映してください。

デプロイスクリプトは、アカウント名、サービス、リージョン、タイムゾーン、タグ設定などの一部をデプロイオプションから自動設定します。これらを変更する場合は、[デプロイ手順](instructions.md)のオプションも確認してください。

### 表示時刻を変更する

```javascript
timezoneOffset: 9,
```

`9` はUTC+9、つまり日本時間です。この値を変更すると、日付の区切り、24時間バー、起動・停止時刻、CloudWatchグラフのタイムゾーンが変わります。

実際の利用拠点のタイムゾーンと一致させてください。

### Nameタグの初期フィルターを変更する

```javascript
tagnameFilter: ".*",
```

`.*` はすべて表示です。

例として本番系だけを初期表示する場合：

```javascript
tagnameFilter: "^prod-",
```

この値は画面右上のNameタグフィルターへ初期入力されます。

### 1ページの件数を変更する

```javascript
tablePageLengthOptions: [10, 20, 50, 100],
defaultTablePageLength: 20,
```

- `tablePageLengthOptions`：画面で選べる件数
- `defaultTablePageLength`：初期件数

`defaultTablePageLength` は `tablePageLengthOptions` に含まれる値にしてください。

### 初期ソートを変更する

```javascript
tableSortOrder: [[1, 'asc'], [0, 'asc']],
```

初期設定では、アカウント列を昇順、その次にNameタグ列を昇順で並べます。

主な列番号：

| 番号 | 列 |
|---:|---|
| 0 | Nameタグ |
| 1 | アカウント |
| 2 | 環境などの分類 |
| 3 | Applicationタグなどのグループ |
| 4 | InstanceType |
| 5 | VPC |
| 6 | AZ |
| 7 | 稼働状況 |
| 8 | 起動～停止 |
| 9 | 起動時間 |
| 10 | 最大CPU |
| 11 | 24時間バー |

Nameタグだけで昇順にする例：

```javascript
tableSortOrder: [[0, 'asc']],
```

### 画面の文言を変更する

```javascript
messages: {
    startstopLoadPossiblyFailed: "一部の稼働履歴を取得できなかったため、表示が正しくない可能性があります。"
},
labels: {
    reloadTableButton: "表示・更新",
    legendStopped: "停止済み",
    legendRunning: "稼働中",
    legendImpaired: "システム異常",
    legendTerminated: "Terminated",
    serviceOptions: [
        { ec2Y_rdsY: "EC2＆RDS" }
    ]
},
```

変更すると、ボタン、凡例、警告メッセージ、EC2/RDS同時表示の選択肢が変わります。

状態判定自体は変わりません。たとえば `legendImpaired` を変更しても、異常と判定する条件は同じです。

### 分類タグと選択肢を変更する

```javascript
categoryTag: {
    label: "環境",
    key: "Env",
    options: [
        { tagValues: ["Production"], display: "本番のみ" },
        { tagValues: ["Development", "Staging"], display: "開発・検証" },
        { tagValues: ["*"], display: "全て(環境タグ無し含む)" }
    ]
}
```

- `label`：画面の列名と選択欄の名前
- `key`：参照するAWSタグキー
- `tagValues`：その選択肢で表示するタグ値
- `display`：ドロップダウンに表示する文言
- `["*"]`：タグなしを含むすべて

複数の `tagValues` を指定すると、いずれかに一致するリソースを表示します。

### URLで絞り込むタグを変更する

```javascript
groupTagFilter: {
    key: "Application",
    value: "",
    keyURL: "application",
    allValue: "ALL"
},
```

- `key`：参照するAWSタグキー。表の列名にも使われる
- `value`：画面を開いたときの初期絞り込み値。空文字なら絞り込みなし
- `keyURL`：URLパラメータ名
- `allValue`：絞り込みを解除する特別な値

たとえば `key` を `System`、`keyURL` を `system` にすると、表の列名は「Systemタグ」になり、次のURLで絞り込めます。

```text
index.html?system=Accounting&loadnow
```

### アカウント、リージョン、RDS表示を変更する

```javascript
accounts: [
    {
        accountName: "本番",
        instanceRegionId: 0,
        additionalService: ["RDS"],
        regions: ["ap-northeast-1", "ap-northeast-3"],
        urlRoot: window.location.origin + "/",
        subDir: ""
    }
],
```

| 設定 | 画面への効果 |
|---|---|
| `accountName` | 画面タイトル、アカウント列、アカウント選択に表示 |
| `instanceRegionId` | `regions` のうち最初に選ぶリージョンを0始まりで指定 |
| `additionalService` | `["RDS"]` ならRDSを表示対象に追加。`[]` ならEC2のみ |
| `regions` | リージョン選択に表示するリージョン |
| `urlRoot` | そのアカウントのデータ取得元CloudFront URL |
| `subDir` | デプロイ時に指定したS3サブディレクトリ |

複数アカウントをまとめる設定方法は[デプロイ手順の「マルチアカウント対応」](instructions.md#マルチアカウント対応)を参照してください。

オプションで `hideDropDown: true` を指定すると、そのアカウントをアカウント選択候補から隠せます。ただし、そのアカウントのCloudFront URLを直接開き、基点アカウントになっている場合は表示されます。

`icon` を設定すると、履歴画面のタイトルへ任意のHTMLアイコンを表示できます。信頼できる固定文字列だけを設定してください。

## 8. データの更新間隔

表示はAWSの状態を定期収集した結果です。

| データ | おおよその更新 |
|---|---|
| EC2の起動・停止、システム異常 | 1分ごと |
| RDSの状態 | 2分ごと |
| CPU使用率 | CloudWatchの5分値を使用し、収集処理は10分ごと |
| 現在の稼働状況 | 一覧表示後、および「状況取得」を押した時点 |

Lambda実行時間、CloudWatch側の反映、ネットワーク、ブラウザキャッシュにより、画面反映が遅れる場合があります。

CPUの直近値は後から補完されることがあります。数分前の値が空でも、時間を置いて再表示すると埋まる場合があります。

## 9. 利用上の制約

- PC専用です。
- データ収集開始前の履歴は表示できません。
- EC2のOS再起動は、AWS上で `running` のままなら起動・停止として記録されません。
- 収集間隔より短い状態変化は記録されない場合があります。
- EC2はNameタグが必須です。Nameタグが設定されていないEC2は、一覧・履歴・CPU使用率の収集対象になりません（RDSはNameタグがない場合、DB識別子で表示されます）。
- Nameタグを主な表示名として使うため、同じNameタグを複数台で使う構成では履歴が曖昧になる場合があります。
- Nameタグを変更すると、変更前後で別の履歴として見える場合があります。
- 削除済みリソースの詳細情報は、保存済み情報がないと取得できません。
- 24時間バーは `config.js` の `timezoneOffset` に従います。閲覧PCのタイムゾーンへ自動追従するものではありません。
- ただし「今日」を表示する場合は、その日付の判定にブラウザのタイムゾーンが使用されます。`timezoneOffset` と閲覧PCのタイムゾーンが異なると、日付の境界や今日のバーの現在時刻の位置がずれる場合があります。
- デプロイ時には、収集用の3つのLambda関数に加えて、WebコンテンツをS3へ配置するためのLambda関数 `*-deploy-web-assets` も生成されます。
- ダッシュボードは監視アラームの代替ではありません。通知が必要な障害監視にはCloudWatch Alarm、EventBridge、AWS Healthなどを併用してください。

## 10. 困ったときの確認順

1. ブラウザを再読み込みする。
2. 「表示・更新」または「状況取得」を再実行する。
3. 日付、リージョン、アカウント、EC2/RDS選択、Nameタグフィルターを確認する。
4. Nameタグフィルターを `.*`、分類を「全て」にして再表示する。
5. URLのクエリ文字列を一度外し、通常のCloudFront URLから開く。
6. AWSマネジメントコンソールで対象リソースが存在するか確認する。
7. 管理者がEventBridge Scheduler、Lambdaログ、S3保存データ、CloudFront/APIの状態を確認する。

デプロイや設定変更の詳細は[デプロイ手順](instructions.md)を参照してください。
